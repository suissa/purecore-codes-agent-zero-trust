# Relatório de Configuração do Hot-Reload (`site/`)
Data: 26 de Fevereiro de 2026

## O Que Foi Feito
- **Análise do arquivo base**: Lemos o conteúdo do utilitário `vite.config.ts` localizado no diretório `/site/`.
- **Injeção da configuração explícita de HMR**: Para atender adequadamente à demanda estrita de recarregamento quente sob sistemas de arquivos isolados (como no Windows Subsystem for Linux), o polling nativo e o overlay de Hot Module Replacement via WebSockets constritos no Vite foram forçados.
- **Opções Inseridas**:
  - `strictPort: true` - Para impedir colidências de socket do server.
  - `interval: 100` - Redução o tempo de checagem do *polling* para atualizar assim que um novo frame for repassado da VM de volta para a IDE nativa.
  - `hmr.overlay: true` - Mostrando erros ao cliente final quando há problemas de transpilação.
- O resultado foi logado no diretório base principal com as devidas inclusões no `CHANGELOG.md`.

## Por Que Foi Feito
- De acordo com as diretrizes do projeto, projetos que utilizem **Vite** requerem o hot-reload mandatório, visando otimizar a Developer Experience (DX). Sistemas rodando o Node dentro de uma virtual engine (WSL) falham com freqência utilizando o FSEvents comum do host (Windows). A habilitação do polling garante detecção fiel de qualquer modificação de sintaxe e atualização em tempo perene!
